# P9 PRD Draft: Chatbox-Native 求职情报与申请包工作台

状态：Draft / 文档审查阶段
日期：2026-07-03
范围：仅定义 P9 目标体验、规格和验收边界，不代表代码实现已经完成。

## 1. 问题定义

当前 P8.1 已经完成 Chatbox-first 自动化候选，具备手动 JD 导入、岗位列表、候选人画像、JD 定制简历、右侧产物台和 provider opt-in 基线。但当前体验仍偏“任务卡片 + 工具入口”组合，用户仍需要理解很多面板和按钮，不能自然地通过 Chatbox 控制完整求职流程。

P9 要解决的问题：

- 用户不想面对大量向导卡片，而是希望直接对 Chatbox 说出求职目标。
- 用户希望系统主动搜索和汇总公开 JD、薪资、城市、技术栈和公司信息。
- 用户希望通过 Chatbox 或 ASR 被引导补全简历、项目故事、能力证据和求职偏好。
- 用户希望右侧产物台持续展示简历、事实摘要、申请包、source refs 和待确认项。
- 用户希望左侧看到招聘市场态势和投递流程，而不是静态任务清单。
- 用户希望顶部快速了解 provider、MCP、Skill、ASR、搜索源和 workspace 状态。

## 2. 产品目标

P9 的目标是把 JobPilot AI 升级为 **Chatbox-native 求职情报与申请包工作台**：

```text
顶部服务中心
左侧求职态势图
中央 Chatbox 主控台
右侧产物台
```

Chatbox 是唯一主控入口。用户通过聊天发起 JD 搜索、招聘信息汇总、资料补全、申请包生成、产物修改和招聘流程更新。左右栏只承担观察、确认和产物展示职责，不抢占中央对话。

## 4. 功能需求

### 4.1 Chatbox 主控

- 用户可以用自然语言发起所有核心操作。
- Chatbox 上方展示紧凑状态机：搜索岗位中、等待补资料、生成申请包、待确认事实、可导出、失败恢复。
- 工具入口以轻量 chips 或命令面板出现，不以大型向导卡片常驻中央。
- 普通聊天不得静默写 artifact；明确生成、更新、确认、导出时才写入产物。

### 4.2 公开 JD 搜索与汇总

- 支持合规公开来源：搜索 API、官方 API、RSS、公司官网公开页面、用户提供 URL。
- 搜索结果标准化为岗位标题、公司、城市、薪资、技术栈、经验要求、source_url、抓取时间和可信度。
- 支持城市、薪资、技术栈、公司阶段和岗位匹配度汇总。
- 第一版不登录招聘平台、不绕反爬、不自动沟通、不自动投递。

### 4.3 Chatbox / ASR 资料采集

- Chatbox 引导用户补充简历模板、项目经历、作品链接、求职偏好和约束条件。
- ASR 作为 opt-in 输入通道，必须显示隐私和外部调用边界。
- ASR 转写内容进入待确认事实，确认后才成为简历或项目故事依据。

### 4.4 多 JD 申请包生成

- 用户可选择不同目标 JD，生成对应简历、项目故事、面试故事、cover letter 或 recruiter message 草稿。
- 每个核心亮点必须有 source refs 或 pending confirmations。
- blocking 待确认项未处理时不得正式导出。

### 4.5 产物对话编辑

- 用户可通过 Chatbox 修改简历、项目事实、表达风格、量化指标和故事结构。
- 系统必须保存版本和变更说明。
- 涉及事实强度变化时必须进入待确认或降级为保守表述。

### 4.6 顶部服务中心

- 展示 provider、MCP、Skill、ASR、搜索源、workspace、诊断状态。
- 配置状态不等于调用成功；真实外呼需要 consent 和 evidence。
- 不展示 API Key、完整 prompt、完整真实资料或 provider raw response。

### 4.7 左侧求职态势图

左侧栏分成三大板块，详见 `09_LEFT_INTELLIGENCE_PANEL_DETAIL.md`。

1. 岗位市场态势：展示公开 JD 搜索后的城市薪资、岗位数量、技术栈热度、来源可信度和更新时间。
2. 目标机会与匹配态势：展示重点岗位、匹配摘要、关键差距、证据缺口和下一步动作。
3. 投递流程态势：展示公司、岗位、阶段、下一步、申请包版本、最近更新时间和 Chatbox 更新记录。

图表必须可折叠，不能挤压中央 Chatbox。用户可以通过 Chatbox 更新流程状态、筛选岗位、要求解释差距或发起申请包生成。

地图类可视化要求：

- 支持地图图钉、拖动、放大、缩小和视角切换。
- 支持三个地图视角：岗位市场、目标机会、投递流程。
- 图钉大小可表示岗位数量，颜色可表示薪资区间、匹配状态或流程阶段。
- 图钉点击后必须解释数据来源和下一步动作。
- 第一版可以使用离线 SVG 原型；生产实现再评估 MapLibre、Leaflet、ECharts 等开源方案。

### 4.8 右侧产物台

- 展示当前目标 JD、候选人事实摘要、CandidateProfile、简历草稿、申请包、source refs、pending confirmations、版本和导出入口。
- 空态必须解释下一步，不出现大面积空白。

## 5. 非目标

- 不登录 BOSS、猎聘、拉勾、LinkedIn 等招聘平台账号。
- 不绕过验证码、反爬、账号权限或平台风控。
- 不自动沟通招聘方，不自动投递。
- 不默认外呼真实 LLM、ASR 或搜索 API。
- 不把 ASR 转写内容直接写成事实。
- 不把无 source refs 的履历内容写入正式简历。
- 不实现 SaaS、多租户、Billing 或公开云同步。

## 6. 成功标准

- 用户首屏能判断：中央 Chatbox 是主控，左侧是态势，右侧是产物，顶部是服务状态。
- 用户无需理解内部 workspace / artifact / job / resume_version，就能通过聊天发起完整流程。
- 至少一条端到端路径可被后续自动化验收：搜索 JD → 汇总薪资城市 → 选择目标岗位 → 补资料 → 生成申请包 → 修改事实 → 更新投递状态。
- HTML 验收报告必须包含真实截图、目标设计图、PRD 追踪和未实现范围。

## 7. 关键风险

| 风险 | 处理原则 |
| --- | --- |
| 全网搜索被误解为招聘平台自动接入 | 只允许合规公开来源；平台登录、绕风控和自动投递单独立项。 |
| ASR 把错误转写写成事实 | ASR 结果必须进入 pending confirmations。 |
| 真实 provider 被默认调用 | provider / ASR / 搜索 API 均需 opt-in。 |
| 图表成为新的复杂面板 | 左侧图表必须折叠，Chatbox 永远优先。 |
| 目标设计图被误认为已实现 | 所有目标图必须标注“目标设计，不代表已实现”。 |
