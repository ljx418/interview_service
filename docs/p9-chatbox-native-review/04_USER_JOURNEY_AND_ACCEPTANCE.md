# P9 用户路线与验收路径

## 1. 端到端路线

```text
搜索 JD
→ 汇总薪资城市
→ 选择目标岗位
→ 补全资料
→ 生成申请包
→ 调整事实和表达
→ 更新投递流程
```

## 2. 路线 1：搜索 JD 并汇总市场

用户输入：

```text
帮我找上海/杭州 25k-40k React + TypeScript 岗位，并按薪资、城市和公司阶段汇总。
```

系统动作：

- 生成搜索任务。
- 读取合规公开来源或 mock source connector。
- 标准化 JD。
- 左侧展示薪资城市和技术栈分布。

验收：

- 每条 JD 有 source refs。
- 报告不声明招聘平台登录或自动抓取通过。

## 3. 路线 2：选择目标 JD

用户输入：

```text
挑 5 个最适合我的岗位，并说明差距。
```

系统动作：

- 结合 CandidateProfile 与 JD。
- 右侧展示目标 JD、匹配摘要和短板。

验收：

- 推荐理由可追溯。
- 缺证据内容进入待确认。

## 4. 路线 3：Chatbox 补全资料

用户输入：

```text
我做过一个后台权限系统，主要负责前端架构和性能优化。
```

系统动作：

- 提取项目故事候选。
- 追问本人贡献、技术难点、结果证据。
- 写入 pending confirmations。

验收：

- 不把用户一句话直接写成正式事实。
- 右侧显示待确认项。

## 5. 路线 4：ASR opt-in 采集项目故事

用户动作：

```text
打开语音补充项目经历。
```

系统动作：

- 明确提示 ASR provider / 本地模式 / 隐私边界。
- 转写片段进入待确认事实。

验收：

- 没有用户确认不得开启 ASR。
- 转写内容不直接进入正式简历。

## 6. 路线 5：生成申请包

用户输入：

```text
给 A 公司生成一版偏前端性能优化的简历和面试故事。
```

系统动作：

- 生成简历、项目故事、面试故事。
- 右侧展示版本、source refs、pending confirmations 和导出预检。

验收：

- blocking 确认项未处理时不得正式导出。

## 7. 路线 6：更新招聘流程

用户输入：

```text
把 B 公司状态改成一面待约，下周三前提醒我补充低代码项目细节。
```

系统动作：

- 更新投递流程图。
- 记录下一步行动。
- 右侧显示关联岗位和备注。

验收：

- 状态变化可见。
- 不自动联系招聘方。
