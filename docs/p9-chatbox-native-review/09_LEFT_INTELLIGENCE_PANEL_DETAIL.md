# P9 左侧求职态势模块详细设计

状态：Draft / 文档审查阶段
用途：细化 P9 左侧栏，不进入代码实现。

## 1. 总体定位

左侧栏不是向导卡片区，也不是表单区。它是 **求职态势观察与流程导航区**，帮助用户回答三个问题：

1. 当前市场上有哪些机会？
2. 哪些机会最值得我优先处理？
3. 我现在投递到了哪个阶段，下一步做什么？

左侧栏必须可折叠，不能抢占中央 Chatbox。任何更新都优先通过 Chatbox 发起，例如：

```text
帮我看上海 React 岗位薪资分布
把 A 公司设为重点目标
把 B 公司状态改成一面待约
提醒我补充 C 公司需要的低代码项目故事
```

## 2. 三大板块

```text
Left Intelligence Panel
├─ A. 岗位市场态势
├─ B. 目标机会与匹配态势
└─ C. 投递流程态势
```

每个板块都至少提供两类可视化：一个地图/空间视角，一个结构化分析视角。地图视角用于快速理解城市、公司和机会分布；结构化视角用于解释原因和下一步动作。

## 3. A. 岗位市场态势

### 3.1 用户问题

- 哪些城市机会最多？
- 哪些薪资段更符合我的预期？
- React / TypeScript / Node / AI 应用等技术栈在 JD 中出现频率如何？
- 哪些岗位来自公司官网、搜索 API、用户粘贴或其他来源？

### 3.2 数据来源

| 数据 | 来源 | P9 状态 |
| --- | --- | --- |
| 标准化 JD | `NormalizedJobPost` | 待开发 |
| 搜索任务 | `JobSearchRun` | 待开发 |
| 用户手动粘贴 JD | 当前 `job` / `source_url` | 已有基线 |
| source refs | JD source_url / import_method | 已有基线，待扩展 |

### 3.3 UI 设计

默认展示为一个紧凑图表页签：

- 顶部摘要：`本轮收集 42 个岗位 / 5 个城市 / 6 个主要技术栈`
- 地图图钉：城市为图钉，图钉大小表示岗位数，颜色表示薪资中位数区间。
- 城市薪资图：城市为横轴，薪资中位数或区间为纵轴。
- 技术栈热度：条形图或 chip cloud，显示 must-have / nice-to-have 频率。
- 来源可信度：公司官网、搜索 API、用户粘贴、其他来源分组。
- 更新时间：显示最近一次搜索任务时间。

### 3.4 交互

- 点击城市：Chatbox 输入区自动填入“分析这个城市的岗位差异”。
- 点击技术栈：右侧产物台筛选相关 JD。
- 点击来源：显示 source refs，不能自动打开平台登录页。
- 折叠状态：只显示关键数字，例如“42 JD / 5 城市 / 25k-40k 主区间”。

### 3.5 验收

- 图表中的每个聚合数字能追溯到 JD source refs。
- 不把搜索源 configured 写成真实搜索已完成。
- 不声明招聘平台登录、自动抓取或自动投递。

## 4. B. 目标机会与匹配态势

### 4.1 用户问题

- 哪些岗位最适合我？
- 哪些岗位薪资高但差距大？
- 哪些岗位应该优先生成申请包？
- 我缺哪些事实、项目故事或能力证据？

### 4.2 数据来源

| 数据 | 来源 | P9 状态 |
| --- | --- | --- |
| 候选人画像 | `candidate_profile` | 已有基线 |
| 能力矩阵 | `skill_evidence` / profile | 已有基线 |
| 岗位匹配 | `match_report` | 已有基线 |
| 多 JD 排序 | `JobMarketInsight` + `match_report` | 待开发 |
| 待补资料 | `pending_confirmations` | 已有基线，待强化 |

### 4.3 UI 设计

展示为“机会优先级列表”：

- 地图图钉：目标城市或公司位置用图钉表示，图钉颜色对应高匹配、需补证据、暂缓。
- 每个机会卡显示：公司、岗位、城市、薪资、匹配度、主要差距、下一步。
- 用颜色表达状态：
  - 绿色：高匹配，可生成申请包。
  - 琥珀色：有机会，但缺项目故事或证据。
  - 红色：关键 must-have 缺失或事实风险高。
- 顶部显示推荐策略：`优先处理 3 个岗位 / 2 个需要补证据 / 1 个不建议投递`。

### 4.4 交互

- 点击“设为重点”：更新当前目标 JD，不立即生成申请包。
- 点击“补证据”：Chatbox 发起追问，例如“请补充你在性能优化中的本人贡献”。
- 点击“生成申请包”：进入右侧产物台生成草稿和待确认项。
- 用户也可通过 Chatbox 说“把这家公司设为重点目标”完成同样动作。

### 4.5 验收

- 推荐理由必须来自 candidate_profile、match_report 或 source refs。
- 匹配度不能作为黑盒分数，必须展示关键依据。
- 缺证据内容不能直接进入正式简历。

## 5. C. 投递流程态势

### 5.1 用户问题

- 我投递了哪些公司？
- 每家公司在什么阶段？
- 下一步要准备什么？
- 哪些岗位已经过期、需要跟进或不再处理？

### 5.2 数据来源

| 数据 | 来源 | P9 状态 |
| --- | --- | --- |
| 投递流程项 | `ApplicationPipelineItem` | 待开发 |
| 当前目标 JD | `job.current_target` 或等价字段 | 已有基线 |
| 申请包版本 | `resume_version` / `application_package` | 已有基线 |
| 用户聊天更新 | `chat_message` + tool intent | 待开发 |
| 下一步提醒 | workspace-local reminder metadata | 待开发 |

### 5.3 UI 设计

投递流程以横向或纵向流程图表达：

```text
收藏 / 待评估
→ 已生成申请包
→ 已投递
→ 筛选中
→ 一面待约
→ 面试中
→ Offer / 关闭
```

同时提供地图视角：

- 按城市展示当前流程阶段分布，例如北京 2 个筛选中、上海 1 个一面待约、杭州 1 个待投递。
- 同一城市多个公司以聚合图钉显示，展开后显示公司清单和阶段颜色。
- 地图图钉点击后，右侧产物台定位到对应公司申请包。

每家公司显示：

- 公司名、岗位名、城市、薪资范围。
- 当前阶段和颜色。
- 下一步行动。
- 关联申请包版本。
- 最近更新时间。

### 5.4 交互

- 用户说“把 B 公司改成一面待约”，流程图更新。
- 用户说“C 公司不考虑了”，状态进入关闭并要求原因。
- 用户说“提醒我周三补低代码项目”，创建本地提醒元数据。
- 点击流程项时，右侧产物台显示该公司的 JD、申请包和待确认项。

### 5.5 验收

- 状态更新可通过 Chatbox 完成。
- 流程变化有时间戳和来源。
- 不自动联系招聘方。
- 不自动投递。
- 不把本地提醒写成外部日历同步已完成。

## 6. 响应式规则

| 视口 | 左侧行为 |
| --- | --- |
| 1440px / 1920px | 三大板块以页签 + 折叠卡片展示，默认展开当前用户最相关板块。 |
| 1200px | 只显示摘要数字和当前选中页签，详细图表可展开。 |
| 720px | 左侧变为顶部/底部页签，默认隐藏，Chatbox 优先。 |
| 390px | 只显示一个“态势”入口，点击进入全屏抽屉。 |

## 7. 与中央 Chatbox 的关系

左侧所有操作都应能被 Chatbox 发起或解释：

- 图表点击只生成意图，不直接执行高风险操作。
- 状态更新必须在 Chatbox 留下可读记录。
- 如果数据不足，Chatbox 负责解释缺少哪些资料。
- 左侧不承载大表单，不要求用户先填完表才能对话。

## 8. False-green 边界

- 市场态势图不得被写成真实招聘平台接入结论。
- 搜索源配置不代表搜索已执行。
- 投递流程更新不代表自动投递或自动沟通。
- 提醒元数据不代表外部日历或消息平台已同步。
- ASR 转写不代表事实已确认。
