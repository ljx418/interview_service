# JobPilot AI P4 UX 体验优化开发及验收计划

日期：2026-06-16
阶段：P4 当前阶段 / 已启动
来源：P3 人工审查反馈

关联审查报告：

```text
docs/active/stage-reviews/P4_UX_TEAM_AUDIT.md
```

## 1. 背景

P3 已完成本地自动化验收：Chatbox 响应闭环、示例/真实资料模式边界、对话区/推进台分离、响应式截图和 HTML 报告均已落地。

但人工审查结论是：

> 认可当前验收报告的大部分内容，但对当前用户体验不完全认同。后续开发阶段必须增加交互体验优化篇幅。

因此，P4 当前阶段应优先处理“真实用户是否愿意、是否顺手、是否理解下一步”的体验问题，而不是继续扩展 MCP、CLI、ASR 或会议平台。

2026-06-17 的人工审查进一步指出：当前实现虽然修复了部分窄屏和基础交互，但全尺寸桌面下仍可能出现大面积空白，页面像窄屏 Chatbox 停靠在左侧，不能被视为完整求职工作台。因此 P4B 增补目标是：把 1200px、1440px、1600px、1920px 全尺寸桌面体验列为 hard gate，并确保截图脚本不会残留 Chrome viewport emulation，避免污染人工审查。

## 2. P4 UX 目标

P4 UX 的目标不是让页面更复杂，而是让求职工作台更像一个可持续使用的工作流工具：

```text
用户打开 Chatbox
→ 清楚知道当前该做什么
→ 输入或上传后知道系统正在做什么
→ 能快速区分聊天、计划、产物、确认项和导出
→ 能在错误或缺资料时自然恢复
→ 能在窄屏和桌面上稳定完成同一条任务
```

## 3. 体验问题清单

P4 启动前需要重新截图审计，至少覆盖：

- 首屏信息密度是否过高；
- Chatbox 和推进台是否仍有认知负担；
- 快捷任务是否像真实操作入口，而不是临时按钮；
- 产物卡是否过度暴露 JSON / 版本细节；
- 用户是否能理解“示例模式”和“我的资料”的后果；
- provider configured 是否容易误解为“正在外呼”；
- 申请包、面试准备、复盘的下一步是否足够明确；
- 移动端是否只是“不坏”，但仍不好用；
- 空状态是否能引导用户开始，而不是展示工程状态；
- 错误态是否给出真实可执行恢复路径。

## 4.1 Gemini 审查后工作包修正

Gemini 审查给出“有条件通过”，并指出任务入口与对话区割裂、状态栏表达生硬、产物卡按钮主次不清、缺少 loading、移动端堆叠压缩 Chatbox 等问题。P4 工作包据此修正如下：

- P4-UX1：Task Launcher 不再作为独立卡片区，而是并入 Chatbox empty state / suggested prompts；
- P4-UX2：suggested prompt 点击后必须填入 composer 或直接触发对话，并显示 loading / thinking / executing 步骤；
- P4-UX2：缺资料、解析失败、后端错误必须给出恢复 action；
- P4-UX3：产物卡必须区分 primary action 和 secondary action，阻塞项优先显示“补充事实 / 去确认”；
- P4-UX3：待确认项文案要解释求职价值，例如“为了让简历更具说服力，建议补充具体指标”；
- P4-UX4：provider 状态改为“外部模型未调用（隐私安全）/ 外部调用需确认”等用户语言；
- P4-UX5：390px 下 Workbench 收为底部抽屉、折叠面板或次级区域，不能压缩 Conversation；
- P4-UX5A：1200px、1440px、1600px、1920px 下不能出现布局错误造成的大面积空白，不能把窄屏布局放大后停靠在左侧；
- P4-UX5A：桌面工作台必须展示对话任务区、状态指标、快捷任务、推进台摘要和下一步建议；
- P4-UX5B：截图脚本必须隔离或清理 viewport emulation / touch emulation，不能污染人工审查者当前浏览器；
- P4-UX5：segmented mode toggle、状态区和按钮必须补齐 ARIA 状态。

## 5. P4 UX 验收门槛

P4 UX 不通过条件：

- 用户仍然不知道第一步该做什么；
- 对话区和推进台仍让用户混淆；
- 缺资料、错误、处理中状态仍不够清楚；
- 产物卡主要依赖 JSON 才能理解；
- provider 状态被误解为默认外部调用；
- 1200px、1440px、1600px、1920px 下仍有由布局错误造成的大面积空白；
- 自动截图脚本残留 Chrome viewport emulation，导致人工浏览器显示状态不可信；
- 移动端虽然不横向溢出，但核心操作仍费力；
- 报告只展示截图，不说明人工体验问题是否改善。

P4 UX 通过条件：

- 人工审查者认可核心路径的交互体验有明显改善；
- examples 路径和我的资料路径的启动方式清楚；
- 用户能在不读开发文档的情况下完成“导入资料 → 粘贴 JD → 生成申请包 → 查看确认项 → 导出”；
- 用户能在 1200px、1440px、1600px、1920px 桌面宽度下看到完整求职工作台，而不是左侧窄栏和右侧空白；
- 自动化测试、前端 build、Chrome 截图和人工审查记录都通过；
- 未把真实个人资料或真实外部 provider 调用写成未执行的验收。

## 6. 非目标

P4 UX 不默认包含：

- MCP Server；
- CLI；
- ASR / Whisper；
- 会议平台接入；
- 自动海投；
- SaaS 登录 / 多租户 / Billing；
- 发布级复杂设计系统；
- 全量重写前端。

这些能力若要进入 P4，必须重新制定阶段 PRD、目标架构、验收门槛和高风险确认流程。

## 7. 启动审计意见

当前建议：

```text
P3 可以保持“本地自动化验收通过”的结论；
但 P3 不应被描述为“人工体验已认可”；
P4 候选阶段应优先做 UX 体验优化，而不是扩展新入口。
```

P4-M0 需要完成：

- 基于当前 P3 截图的人类体验问题清单；
- P4 UX 设计/交互验收标准；
- P4 UX before/after 截图计划；
- Gemini 前端审查包；
- drawio 目标架构与当前架构差异同步；
- 继续使用 examples 作为主要自动化验收数据，真实个人资料受控体验审查仍需人工确认。

## 8. P4-M0 审计结论

当前 P4 UX 文档体系已经可以支撑本阶段启动：

```text

→ P4 验收门槛
→ P4 自动化范围

→ P4 drawio 图

```

审计意见：

- 可以进入 P4-M1 前端体验优化；

- 不能把静态原型、Gemini 审查意见或未执行的人工体验结论写成已实现。

- 认可 P4B：补齐冻结记录，复跑 pytest、frontend build 和 drawio XML parse，进入 P4 冻结评审；
