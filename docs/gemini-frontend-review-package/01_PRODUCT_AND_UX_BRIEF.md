# 产品与 UX Brief

## 产品定位

JobPilot AI 是本地优先的求职 Agent 工作台，面向正在转行成为程序员的人。用户通常有项目经历，但不知道如何把项目、技能、JD 和面试材料组织成可信的求职产物。

核心入口是 Chatbox，但产品不应像普通闲聊机器人。它应像一个“求职材料工作台”：

```text
导入资料
→ 分析岗位
→ 生成申请包
→ 确认和编辑
→ 导出材料
→ 面试准备和复盘
```

## 目标用户

- 转行程序员，正在申请 junior / entry-level 软件岗位；
- 有项目 README、简历或作品，但缺少求职表达结构；
- 重视本地隐私，不希望默认把个人资料发给外部模型；
- 希望 AI 生成的内容能说明来源、待确认风险和可导出结果。

## 当前 UX 问题

- 首屏仍像工程状态页，任务入口不够自然；
- Chatbox、计划、推进台、产物卡之间的边界仍有认知负担；
- 产物卡过多暴露 artifact、version、JSON 等工程字段；
- provider 状态容易让用户误解是否已经外呼；
- 移动端能跑，但任务层级和滚动体验仍不够顺手；
- 验收报告已覆盖截图，但人工体验仍未完全认可。

## P4 UX 目标

1. 用户 5 秒内知道第一步。
2. 用户发送任务后能看懂系统在做什么。
3. 用户能区分对话区和求职推进台。
4. 用户能不看 JSON 就理解产物价值、风险和下一步。
5. 用户能理解示例模式、我的资料模式、mock provider、external provider 的差异。
6. 1280px、720px、390px 下都能完成关键路径。

## Gemini 第一轮审查后采纳的修正

- 任务入口必须并入 Chatbox 空状态 suggested prompts，不再作为割裂的独立任务卡区；
- suggested prompt 点击后要填入输入框或直接触发对话；
- provider 状态使用“外部模型未调用（隐私安全）”等用户语言；
- 对话区需要 loading / thinking / executing 状态；
- 错误态需要恢复 action；
- 产物卡需要 primary / secondary action 层级；
- 390px 下 Workbench 应折叠为底部抽屉或次级面板；
- segmented 控件和状态区需要 ARIA 状态。

## 非目标

- 不做营销首页；
- 不做复杂 dashboard；
- 不引入 SaaS 登录、多租户、Billing；
- 不做 MCP、CLI、ASR、会议平台；
- 不做自动海投；
- 不默认启用真实外部 provider；
- 不用静态原型替代真实实现验收。
