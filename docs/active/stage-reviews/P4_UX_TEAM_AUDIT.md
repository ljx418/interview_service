# P4 UX 团队审查报告

日期：2026-06-16  
范围：当前 P3 Chatbox 工作台、P3 截图证据、前端实现、P4 UX 计划  
审查视角：UX Architect、UI Designer、Product Manager、Accessibility Auditor  
结论：当前 P3 自动化验收成立，但用户体验、视觉高级感和交互易用性尚不足以进入“人工体验认可”状态。P4 应优先做 UX 体验优化，不建议先扩展 MCP、CLI、ASR 或会议平台。

## 1. 总体判断

当前界面已经能证明工程闭环：

- Chatbox 有响应；
- 示例模式 / 我的资料模式边界可见；
- 对话区和推进台已分离；
- 1280 / 720 / 390 截图可用；
- pytest 与前端 build 通过。

但从真实用户体验看，当前仍像“工程验收台”，不是“求职材料工作台”。主要问题是：用户进入页面后看到大量状态、模式、流程、产物、版本、运行边界，但不知道最应该关注哪一个动作、哪一个结果、哪一个风险。

P4 的核心目标应改为：

```text
减少工程状态暴露
→ 强化任务入口
→ 让产物更像可用求职材料
→ 让确认 / 编辑 / 导出成为自然下一步
→ 让移动端真正顺手
```

## 2. 高优先级问题

### 2.1 首屏主任务不够突出

严重度：高  
视角：Product Manager / UX Architect  
位置：桌面和移动初始态

现状：

- 首屏同时出现 header 状态、模式切换、快捷任务、对话消息、推进台、运行边界；
- “先上传简历或项目 README，再粘贴 JD”只是普通消息卡；
- 用户很难一眼判断第一步到底是上传文件、跑示例、切换模式，还是点击快捷任务。

影响：

- 新用户启动成本高；
- Chatbox-first 的定位被“推进台 + 状态面板”稀释；
- 用户可能先点“申请包”，然后收到错误，再理解流程。

建议：

- 首屏引入明确的任务启动区：`上传资料`、`粘贴 JD`、`跑示例` 三个主入口；
- 把“生成申请包 / 准备面试”降级为资料和 JD 就绪后的下一步；
- 未导入资料前，推进台只显示极简下一步，不展示完整 7 步矩阵。

### 2.2 推进台信息过早、过多

严重度：高  
视角：UX Architect / UI Designer  
位置：右侧 `求职推进台`

现状：

- 用户刚进入页面就看到 7 个阶段、3 个指标卡和“一键体验”；
- 这些内容对验收者有用，但对求职用户来说太像项目管理面板；
- 指标 `0/7`、`待分析`、`0` 比“下一步该做什么”更抢眼。

影响：

- 产品显得复杂；
- 核心体验从“对话生成结果”变成“操作流程板”；
- 移动端需要滚很长一段才看到运行边界和导出状态。

建议：

- 推进台改为渐进披露：
  - 初始：只显示当前阶段和下一步；
  - 有 JD 后：显示岗位分析和匹配；
  - 有申请包后：显示确认项、版本和导出；
- 7 步流程改成折叠的“查看完整流程”；
- 移动端默认折叠推进台，只保留当前下一步。

### 2.3 产物卡仍偏工程对象，不像求职材料

严重度：高  
视角：UI Designer / Product Manager  
位置：`ArtifactCard`

现状：

- 产物类型显示 `job`、`match_report`；
- 状态显示 `needs_confirmation`、`current artver_...`；
- 操作按钮只有图标，依赖 title tooltip；
- JSON 折叠入口仍很靠前。

影响：

- 转行求职用户不理解 artifact、version、needs_confirmation；
- 用户不确定“确认”是在确认什么；
- 产物卡没有把“可直接使用的内容”放在最前面。

建议：

- 面向用户重命名：
  - `job` → `岗位解析`;
  - `match_report` → `匹配报告`;
  - `application_package` → `申请包`;
  - `needs_confirmation` → `待确认`;
- 版本号默认弱化，只在“版本历史”里显示；
- 操作按钮增加短标签或分组：`确认事实`、`编辑`、`重新生成`、`导出`;
- JSON 移入“开发信息”或“高级信息”，默认不作为普通用户路径的一部分。

### 2.4 示例模式 / 我的资料模式语义仍不够安全

严重度：中高  
视角：Product Manager / Accessibility Auditor  
位置：Header、Chat controls、运行边界

现状：

- Header 显示 `External configured`，视觉权重接近 `Workspace ready`；
- 用户可能误解为系统正在使用外部模型；
- `示例模式` 和 `我的资料` 是按钮，但切换后没有明确解释“会影响哪些数据源和调用行为”。

影响：

- 隐私感知不稳定；
- 用户可能不敢上传真实资料；
- provider configured / called 的差异仍需要读说明文字才能理解。

建议：

- 把 provider 状态改成更明确的三态：
  - `本地 mock`;
  - `外部已配置，未调用`;
  - `外部调用中 / 已调用`;
- 我的资料模式下增加短提示：`仅本地处理，外部调用需二次确认`;
- 示例模式下把“示例数据”作为水印或小标签贯穿 demo 结果，防止混淆。

## 3. 中优先级问题

### 3.1 视觉高级感不足，偏默认 SaaS 骨架

严重度：中  
视角：UI Designer

现状：

- 视觉语言以白卡、蓝色按钮、浅灰背景为主；
- 组件密度和边框数量偏多；
- 左侧深色 sidebar、顶部状态、主卡片、右侧卡片之间的层级没有形成高级产品的“安静秩序”；
- 多数卡片都是同样的 8px 边框和阴影，缺少主次。

建议：

- 建立更克制的 design token：
  - 主背景、工作台背景、内容面板、浮层面板分层；
  - 降低边框数量，增加留白和分组；
  - 减少蓝色高亮使用，只给主 CTA 和当前焦点；
- 对 Chatbox 做“工具型产品”视觉，而不是 marketing 或 dashboard；
- 保留 8px 以下圆角，但统一 card、button、input 的视觉重量。

### 3.2 快捷任务更像按钮堆，不像流程入口

严重度：中  
视角：UX Architect / Product Manager

现状：

- `整理资料 / 申请包 / 面试 / 跑示例` 并列；
- 这些任务有依赖关系，但 UI 没有表达依赖；
- `申请包` 在没有 JD 时仍可点击，只能靠错误提示恢复。

建议：

- 快捷任务按阶段启用：
  - 无资料：上传 / 跑示例；
  - 有资料无 JD：粘贴 JD / 分析岗位；
  - 有 JD：生成申请包 / 准备面试；
- 禁用态要解释原因，例如 `需要先分析 JD`;
- 把“跑示例”从日常任务中分离，放到示例模式启动区。

### 3.3 移动端可用但不顺手

严重度：中  
视角：UX Architect / Accessibility Auditor

现状：

- 390px 下没有严重横向裁切；
- 但首屏之后需要较长滚动才能看到推进台；
- 输入区固定在卡片底部，不是屏幕底部；
- 长用户消息占据大量垂直空间，产物卡内容被推到下方。

建议：

- 移动端增加“当前任务摘要”粘性条；
- 推进台默认折叠；
- 输入区考虑视口底部 sticky，但必须避免遮挡消息；
- 长 JD 消息默认折叠为“已接收 JD，展开查看”。

## 4. 可访问性审查

### 4.1 图标按钮缺少可见名称

严重度：中  
WCAG：2.5.3 Label in Name、4.1.2 Name, Role, Value  
位置：artifact 操作按钮、上传按钮、发送按钮

现状：

- 操作按钮主要依赖 `title`；
- 视觉用户和触屏用户不一定能看到 title；
- 屏幕阅读器用户可能只听到图标上下文不足。

建议：

- 所有图标按钮增加 `aria-label`；
- 关键动作在桌面也显示短文本，至少在 artifact 卡中显示；
- 上传按钮使用真实 button + hidden input，明确 label。

### 4.2 状态更新的 aria-live 粒度需要检查

严重度：中  
WCAG：4.1.3 Status Messages  
位置：消息流、busy 状态、错误态

现状：

- messages 容器有 `aria-live="polite"`；
- 但处理中、错误、产物新增是否被合理朗读仍未做真实 screen reader 验证；
- artifact card 进入消息流后信息量很大，可能造成朗读负担。

建议：

- 对短状态使用单独 `role="status"`；
- 对错误使用 `role="alert"` 或明确错误区域；
- artifact card 增加摘要化可读文本，避免直接朗读大量结构化信息。

### 4.3 色彩和状态不能只依赖颜色

严重度：中  
WCAG：1.4.1 Use of Color  
位置：status pill、version、confirmation

现状：

- 绿色 / 黄色承担大量语义；
- 文本存在，但部分英文状态仍不易理解。

建议：

- 状态统一中文化；
- 给状态增加图标和明确动词，例如 `待确认 2 项`、`外部已配置，未调用`。

## 5. 团队视角结论

### UX Architect

当前架构分层已经成立，但信息架构仍没有围绕用户主任务组织。P4 应优先重排页面层级：启动任务、当前上下文、产物、确认项、导出，而不是继续展示完整工程流程。

### UI Designer

当前界面合格但不高级。高级感的关键不是更多装饰，而是更少的边框、更清晰的主次、更稳定的留白、更少的状态噪音，以及对产物内容的精心排版。

### Product Manager

当前产品证明“能跑”，但还没证明“用户愿意用”。P4 成功指标不应只是截图通过，而应是用户能在不读文档的情况下完成申请包路径，并能说清楚下一步。

### Accessibility Auditor

当前具备部分语义基础，但还没有做真实键盘和屏幕阅读器审计。P4 不能只做视觉精修，需要把图标按钮名称、状态朗读、错误恢复和触控目标作为验收门槛。

## 6. P4 建议优先级

### 第一优先级：任务启动体验

- 明确首屏三入口：上传资料、粘贴 JD、跑示例；
- 隐藏或折叠非当前阶段信息；
- 根据上下文启用任务按钮。

### 第二优先级：产物卡重设计

- 用户语言替代工程字段；
- 摘要、确认项、导出放前面；
- JSON 和版本历史后置。

### 第三优先级：状态和隐私表达

- provider 状态改为 configured / not called / calling / called；
- 示例数据贯穿标识；
- 我的资料模式强化本地处理和二次确认。

### 第四优先级：移动端交互精修

- 折叠推进台；
- 长 JD 折叠；
- 输入区与当前任务保持可见；
- 390px 重新做 before/after 人工审查。

### 第五优先级：可访问性验收

- 图标按钮 `aria-label`；
- keyboard-only 完成核心路径；
- VoiceOver 冒烟测试；
- 200% zoom 和 reduced motion 检查。

## 7. 建议的 P4 UX 出门条件

P4 UX 不能只通过自动化截图。建议出门条件为：

- `python3 -m pytest` 通过；
- `npm --prefix apps/chatbox run build` 通过；
- 1280 / 720 / 390 before/after 截图；
- 人工审查确认核心路径体验有明显改善；
- 用户不读文档也能完成：

```text
上传资料
→ 粘贴 JD
→ 生成申请包
→ 查看待确认项
→ 导出
```

- 报告中明确列出仍未完成的体验问题，不做虚假验收。
