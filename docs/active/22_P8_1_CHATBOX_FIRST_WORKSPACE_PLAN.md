# P8.1 Chatbox-first 工作台信息架构修正计划

## 1. 阶段状态

P8.1 当前处于文档开发阶段，尚未进入前端或后端代码实现。P8.1 不是推翻 P8-JD Intake 与简历生成能力，而是在 P8 自动化候选完成后修正体验主次关系。

P8 已经完成资料准备向导、JD 手动导入中心、多 JD 目标岗位、JD 定制简历、source refs、pending confirmations、export preflight、专项 eval 和中文 HTML 截图报告。当前问题是这些入口在界面中占据了 Chatbox 中央区域的首屏优先级，让用户感觉聊天框被弱化或消失。

P8.1 的核心目标：

```text
用户指导 - Chatbox - 工作台
```

其中 Chatbox 必须始终是用户第一眼可见、第一优先交互、第一默认路径。资料准备、JD 导入、岗位选择和简历生成应作为聊天驱动的辅助面板和工作台能力出现，而不是把大型表单放在聊天时间线之前。

## 2. 问题定义

当前 P8 体验存在以下信息架构风险：

- 中央对话区顶部展示较重的 P8 workflow strip，包含资料准备、JD 导入、岗位列表和简历生成入口；
- 聊天时间线被推到下方，首屏心理模型从“我和求职 Agent 对话”变成“我先填多个表单”；
- 左侧用户指导、中央聊天、右侧工作台三栏关系不够清晰；
- 用户无法稳定判断哪些内容是指导、哪些内容是对话、哪些内容是产物；
- 资料/JD/简历生成虽然功能存在，但没有通过 Chatbox 形成自然、连续、低压力的用户引导。

必须避免的错误方向：

- 把 Chatbox 做成隐藏在任务流之后的次级控件；
- 把五类资料、JD 表单和简历生成全部堆在中央首屏；
- 用更多按钮解决用户不知道提供什么资料的问题；
- 把 P8.1 写成招聘平台自动接入、真实 provider 质量验收或真实个人资料路径验收。

## 3. 目标体验

P8.1 完成后，用户打开本地 Chatbox 的目标体验应为：

```text
打开页面
→ 左侧看到“我该准备什么”的轻量用户指导
→ 中央第一优先看到 Chatbox 对话、Agent 当前状态和输入框
→ 用户可以直接问“帮我生成一版前端岗位简历”
→ Agent 在对话中提示缺少哪些资料，并给出可点的资料补充入口
→ 用户从输入框上方的紧邻工具条上传资料、粘贴 JD、选择目标岗位
→ 右侧工作台展示岗位、画像、简历草稿、source refs、待确认项和导出前检查
→ 用户围绕同一个 Chatbox 连续追问、补资料、确认事实和导出
```

三栏职责必须稳定：

| 栏目 | 优先级 | 职责 | 不允许承担的职责 |
| --- | --- | --- | --- |
| 用户指导 | 辅助 | 解释需要哪些资料、缺失影响、下一步建议、示例资料入口 | 不承载大型编辑表单；不替代聊天 |
| Chatbox | 主路径 | 连续对话、Agent 状态、输入框、资料/JD 快捷入口、意图触发 | 不被 workflow strip 或大表单挤出首屏 |
| 工作台 | 产物 | 岗位列表、当前目标 JD、画像、简历草稿、source refs、pending confirmations、export preflight | 不要求用户先理解内部 artifact 结构 |

## 4. 目标信息架构

### 4.1 左侧：用户指导

左侧应回答“我需要提供什么”和“下一步做什么”：

- 当前任务：如“准备求职材料”“完善目标 JD”“确认简历事实”；
- 资料清单：简历、项目经历、作品链接、目标 JD、求职偏好；
- 缺失影响：缺少项目经历会影响项目描述，缺少 JD 会影响定制化；
- 示例入口：可运行示例路径，但必须标注示例不等于真实资料验收；
- 下一步建议：由当前 workspace 状态驱动，而不是静态按钮堆叠。

### 4.2 中央：Chatbox

中央必须是可连续使用的 Chatbox：

- 顶部只保留紧凑 Agent 状态机：空闲、等待资料、解析中、生成中、待确认、可导出；
- 聊天时间线在首屏中部可见；
- 输入框始终稳定可见或在视口底部可达；
- 输入框上方紧贴一排小型工具入口：上传资料、粘贴 JD、选择目标岗位、生成简历；
- 工具入口打开轻量弹层、抽屉或右侧工作台面板，不把大型表单直接插到时间线前；
- 普通聊天不静默写 artifact；明确生成、刷新、导出时才进入工具路径。

### 4.3 右侧：工作台

右侧承载产物和确认：

- 当前目标岗位；
- 多 JD 列表和匹配摘要；
- CandidateProfile 摘要；
- JD 定制简历草稿；
- source refs；
- pending confirmations；
- export preflight；
- 最新 artifacts 和下载入口。

右侧应支持空态、生成中、待确认、可导出和失败恢复状态，避免用户看到空白大面板。

## 5. 响应式原则

桌面端：

- 1200px、1440px、1600px、1920px 下保持三栏结构；
- 中央 Chatbox 最小宽度优先保障；
- 左侧和右侧可压缩，但不能压缩到导致文字重叠；
- 首屏必须看到聊天时间线和输入框，或看到清晰的输入框固定区域。

平板端：

- 默认保留 Chatbox 主视图；
- 用户指导和工作台可变为顶部摘要、侧边抽屉或标签页；
- 不允许让资料表单占满首屏并隐藏聊天。

移动端：

- 默认显示 Chatbox；
- 用户指导、资料清单、JD 列表和工作台通过抽屉或分段切换进入；
- 输入框、上传资料、粘贴 JD、发送按钮必须可达且不重叠。

## 7. 验收门槛

P8.1 必须通过以下门槛后，才可以写成自动化候选完成：

- 首屏主路径是 Chatbox，而不是资料/JD/简历生成大表单；
- 页面结构能被用户快速理解为“用户指导 - Chatbox - 工作台”；
- 上传资料、粘贴 JD、选择目标岗位、生成简历入口紧贴输入框或在清晰辅助区域内；
- 中央对话区保留连续多轮对话能力，不因工具入口切换而丢失上下文；
- Agent 状态机显示当前执行状态：等待资料、解析 JD、生成简历、待确认、可导出、失败恢复；
- 右侧工作台显示产物和确认项，不抢占聊天主路径；
- 1200px、1440px、1920px、720px、390px 截图无按钮错位、文字重叠或核心入口不可达；
- 报告不得声称 BOSS/招聘平台自动接入、真实 provider 质量、真实个人资料路径、自动投递、SaaS、ASR 或会议平台已通过。

## 8. 高风险边界

P8.1 不改变以下边界：

- 不登录招聘平台；
- 不抓取 BOSS、猎聘、拉勾或其他平台页面；
- 不自动沟通或自动投递；
- 不默认调用真实 MiniMax、DeepSeek 或 OpenAI-compatible provider；
- 不读取未授权真实个人资料；
- 不保存 API Key 到前端、仓库、日志、报告或截图；
- 不执行 workspace 删除、迁移 apply 或不可逆操作。

## 9. 文档支撑结论

P8.1 文档目标明确、边界清楚、验收门槛可自动化验证。当前可以进入 P8.1 文档审计和人工方向审核；在用户认可后，再进入 P8.1-M0 开发前启动审计和后续自动化开发。

外部审计意见已采纳：当前 P8.1 文档体系已经能支撑 P8.1-M0 到 P8.1-M5 按阶段开发、验收和打回；不建议继续扩写主设计文档。后续动作应从 P8.1-M0 开发前启动审计开始，而不是继续增加规划文件。
