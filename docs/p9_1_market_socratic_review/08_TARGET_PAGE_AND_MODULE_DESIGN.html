3.0 开源地图体验参照与备选方案

本节暂停当前自制方案，先审查成熟开源项目的交互模式。仓库信息通过 GitHub CLI 核验；以下内容是方案参照，不代表本项目已经引入依赖或完成代码实现。

Apache ECharts 首选

apache/echarts66.7k starsApache-2.0

github.com/apache/echarts

可借鉴体验：`visualMap` 解释颜色语义，`tooltip` 做城市/区域浮层，`toolbox` 提供缩放/还原/导出证据，`geo + scatter` 同时表达行政区划和城市岗位量。
适配判断：最适合 JobPilot 左侧情报面板。风险是 GeoJSON 质量和 ECharts option 复杂度，需要明确 fallback 和 source refs。

Kepler.gl 交互参考

keplergl/kepler.gl11.9k starsMIT

github.com/keplergl/kepler.gl

适配判断：适合借鉴“图层面板 + 地图探索”体验，不建议直接嵌入，否则 JobPilot 会膨胀成 GIS/BI 工具。

deck.gl 高级视觉

visgl/deck.gl14.3k starsMIT

适配判断：视觉上限高，但对 P9.1 左侧面板偏重。更适合后续真实数据规模变大后作为二期路线。

AntV L7 国产地理可视化

antvis/L74.0k starsMIT

github.com/antvis/L7

可借鉴体验：polygon/point/heatmap 图层、WebGL 地理分析、地图控件、城市级下钻和更精致的空间视觉。
适配判断：适合中国语境和 GeoJSON 图层，但依赖栈比 ECharts 更重，首版应作为备选而不是默认路线。

MapLibre GL JS 真实地图路线

maplibre/maplibre-gl-js11.0k starsBSD-style

github.com/maplibre/maplibre-gl-js

可借鉴体验：真实矢量瓦片地图、流畅 zoom/pan、地图控件、地理定位式探索。
适配判断：用户已明确更偏行政区划态势图，不是街道导航地图；仅在未来需要真实底图时使用。

react-simple-maps 轻量 SVG

zcreativelabs/react-simple-maps3.3k starsMIT

github.com/zcreativelabs/react-simple-maps

可借鉴体验：React 声明式 SVG 地图、d3-geo/topojson 投影，适合轻量行政区划图和自定义交互。
适配判断：比手写 SVG 好，但图层、tooltip、visualMap、统计图联动都要自己补，整体不如 ECharts 省力。
应借鉴 1：图层面板像 Kepler.gl 一样把机会、薪资、技术栈、来源可信度做成可切换图层，而不是把所有信息塞进地图。
应借鉴 2：语义图例像 ECharts visualMap 一样让颜色有明确含义，并始终可见，避免用户误读样例数据。
应借鉴 3：探索反馈hover 预览、click 选中、右侧详情同步、Chatbox 草稿生成要形成完整反馈链。
应避免：BI 工具化不要照搬 Kepler/Superset/Grafana 的完整数据平台体验，JobPilot 的主路径仍是 Chatbox。

确定主路线

`ECharts map/geo + 合法 GeoJSON + visualMap + scatter + drilldown`。默认左侧面板是二维行政区划情报图，不是街道瓦片地图。
`globe.gl` 只作为展开态“大地图/地球仪感”的交互参考，不替代行政区划下钻主图。

下钻状态机

全国 → 省/直辖市 → 城市 → 区县。每次点击更新 `current_adcode`、`level`、`breadcrumb`、`selected_region`、`hover_region`、`fallback_state`。
下钻失败、GeoJSON 缺失或许可不清时必须显示 fallback，不能空白通过，也不能静默改用 fixture 后宣称真实市场。

力感反馈

hover 区域抬升/描边/tooltip，click 后区域加深、城市气泡放大、详情面板同步、Chatbox 草稿出现，breadcrumb 可返回。
反馈只证明交互设计，不证明真实 provider、招聘平台抓取或真实市场数据已经接入。

4. 前端模块详细设计图

4.0 GitHub CLI 地图库调研与选型

经过重新评估，P9.1 不再采用 Leaflet/OSM 真实地图作为主方案。左侧模块目标是“求职态势理解”，不是导航定位，因此主方案改为 ECharts map/geo + 合法 GeoJSON 行政区划下钻可视化；真实地图仅作为被否定的备选路线记录。

候选模块适配判断阶段结论
Apache ECharts map/geo + GeoJSON最适合行政区划色块、城市气泡、薪资直方图和技术栈图层，能自然承载求职态势。推荐作为 P9.1 下一阶段主路线。
D3 + GeoJSON / TopoJSON自由度最高，适合变形地图和复杂交互，但工程成本更高。作为高级备选。
Leaflet / MapLibre / OpenLayers更适合真实地理地图、导航、瓦片和 GIS 场景。不作为当前主方案，仅保留备选。

4.1 TopServiceCenter

顶部只解释系统能力状态，不是业务主入口。所有外部能力必须区分未配置、已配置、已连通、已调用、失败和 fallback。

  • 点击配置只打开设置或说明，不默认外呼。
  • 市场 provider 未授权时显示 not_configured。
  • 不得把配置展示写成真实连通验收。

4.2 MarketIntelligenceMap

ECharts 情报板模块结构 ECharts + GeoJSON
组件分层
  • `MarketIntelligenceBoard`：左侧完整求职情报板，保持 Chatbox 辅助角色。
  • `EChartsOptionBuilder`：生成 map/geo、visualMap、scatter、tooltip、toolbox 的 option。
  • `RegionOpportunityMap`：基于 GeoJSON 的行政区划 choropleth。
  • `CityScatterLayer`：重点城市气泡，尺寸表达岗位量，颜色表达薪资或热度。
  • `MarketLayerTabs`：机会热度、薪资中位、技术栈、来源可信度图层。
  • `RegionInsightPanel`：选区详情、薪资分布、技术栈需求和 Chatbox 下一步。
  • `SourceTrustLegend`：fixture / manual / public / opt-in API 固定可见。
  • `ChatboxPromptBridge`：点击城市、薪资段或技术栈后生成可编辑追问草稿。

地图是 ECharts 行政区划求职情报板，不是暗色大屏，也不是普通导航地图。它要服务求职决策：区域机会、薪资层、技术栈热度、来源可信度和 Chatbox 追问。

  • 必须有行政区划色块、城市气泡、visualMap、tooltip、toolbox、图层切换、选区详情和 source filter。
  • 点击省份、城市群或城市后产生 Chatbox 追问上下文，并在右侧产物台记录 market insight。
  • 每个数字有 source refs 或 fixture 标签。
  • 行政区划 GeoJSON 加载失败时必须显示 fallback，不得空白通过。
  • 不得再采用假控制中心暗色皮肤；控件、图例、选区和数据边界必须可读、可验收。

4.3 SocraticChatbox

Chatbox 负责推进事实采集。它不抛出大表单，不一次问多个问题，而是每轮选择一个最能提升事实质量的问题。

  • 状态机从目标岗位到证据检查再到故事草稿。
  • 不确定事实进入 pending confirmations。
  • 禁止编造学历、年限、公司、指标和本人贡献。

4.4 ArtifactWorkbench

产物台不是新的表单入口，而是对话过程中的事实和产物审查区。它让用户随时看到哪些内容可写、哪些还缺证据。

  • source refs 和 pending confirmations 固定可见。
  • Chatbox 修改事实后，产物台同步更新。
  • blocking 待确认项未处理时不得正式导出。

5. 用户操作 / 体验路线图

1. 打开工作台 顶部看到 provider 和安全边界，中央 Chatbox 默认可输入，左侧是市场情报。
2. 查询市场 用户问北京/上海 LLM 前端岗位，系统先说明数据来源和 provider 状态。
3. 操作地图 用户缩放、拖动、点击城市或筛选技术栈，Chatbox 获得上下文。
4. Socratic 补故事 Agent 一问一答采集目标岗位、背景、职责、难点、行动、指标和证据。
5. 审查产物 右侧展示事实摘要、故事草稿、JD 映射、source refs 和待确认项。

7. 设计一致性与事实边界自检

一致性通过项

  • 始终采用顶部服务中心、左侧市场情报、中央 Chatbox、右侧产物台四区结构。
  • 视觉 token 使用克制商务色，加入数据蓝、薪资琥珀、证据紫灰和风险红。
  • 所有目标图均为原型，不写成真实界面验收证据。
  • Chatbox 是主路径，左侧和右侧只做辅助信息与产物审查。

不可声明能力

  • 不得声明 P9.1 UI 已实现或已人工验收。
  • 不得声明真实市场 provider、BOSS/猎聘/拉勾/LinkedIn 抓取已接入。
  • 不得声明真实 ASR、真实 LLM provider、自动投递、MCP/Skill 连通性通过。
  • 不得用 fixture 图表冒充真实市场数据。
