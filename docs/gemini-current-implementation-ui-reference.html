当前产品目标

JobPilot AI 目标是把求职者资料、职位意图、简历产物、模拟面试和后续投递工作流组织到一个连续的 AI 工作台内。 当前阶段聚焦 Chatbox 的连续对话、资料沉淀、工作台可见性和人工体验审查前的自动化证据收口。

当前界面形态

桌面端为三栏工作台：左侧导航和状态，中间对话流，右侧 Workbench 产物区。 窄屏和移动端通过抽屉或压缩布局继续访问会话、任务与产物。

当前主要问题

视觉风格仍显粗糙，卡片密度和层级不够清晰，任务区与对话区的产品感不足。 Artifact 生成后的右侧工作台展示存在局部不一致，需要作为体验缺陷保留。

目标架构与当前实现

以下是给 Gemini 判断“哪些能设计、哪些不能承诺”的事实边界。

架构层 目标架构 当前实现状态 设计时不能越界的点
Experience Shell 统一承载聊天、任务、资料、产物和阶段状态的求职工作台。 Chatbox 前端已具备桌面三栏和移动端压缩路径。 可以优化为更专业的应用工作台，但不要做成营销落地页。
Conversation Plane 支持自然语言、多轮连续对话、阶段引导和自由输入。 当前自动化覆盖了 guided path、free chat 输入和状态查询。 自由 chatbox 和无中断连续多轮能力仍是后续目标，不能写成已完整产品化。
Workbench Plane 展示用户资料、职位意图、简历、职业事实和后续工作流产物。 后端和消息引用能产生 career_facts artifact，右侧 UI 有局部刷新不一致。 必须保留“部分通过/待修复”的表达，不能把缺陷藏掉。
Provider Layer 可接入真实外部模型和未来 ASR、会议、投递平台。 当前验收是本地 mock/demo，不包含真实 API Key 和真实外部调用默认路径。 不要写出任何表示真实 provider、真实个人资料路径已经完成验收的文案。
Evidence & Acceptance 用自动化截图、测试报告和人工体验审查共同收口。 已有 P4 自动化验收报告和截图证据，人工认可未完成。 Gemini 输出只能辅助设计优化，不能替代人工体验验收。

当前可以体验的用户路径

这些路径已经通过自动化截图形成证据，可作为 Gemini 设计原型的页面树基础。

路径 A：打开 Chatbox 工作台

用户进入桌面端工作台，看到侧边栏、对话区、任务状态和右侧产物/资料区域。可作为首屏主界面设计。

路径 B：完成 guided path

用户按引导输入目标信息，系统推进阶段并给出任务反馈。可作为“结构化引导 + 聊天”的核心体验设计。

路径 C：自由输入并查询状态

用户可以输入自由文本、继续第二轮对话，并通过状态查询查看阶段和 workspace 变化。

路径 D：生成并查看 career facts

系统可生成职业事实 artifact，消息流和后端引用存在证据；右侧工作台展示仍有局部不一致。

路径 E：移动端恢复访问

390px 移动视口下，页面可恢复到可访问状态，适合让 Gemini 设计移动端抽屉和压缩面板。

路径 F：多桌面宽度检查

1200、1280、1440、1600、1920px 视口都有截图，适合检查三栏布局比例、信息密度和视觉稳定性。
