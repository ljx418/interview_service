# JobPilot AI 基本能力集 v1.0

版本：v1.0  
日期：2026-06-09  
定位：面向转行程序员的免费开源求职 Agent 服务  
入口：Chatbox-first，本地优先，无注册登录

---

## 1. 能力分层

JobPilot AI 的能力不按页面划分，而按 Agent 可调用能力划分。

```text
用户输入
→ Agent 识别任务
→ 调用领域工具
→ 生成可落地产物
→ 用户确认
→ 保存到本地工作区
```

---

## 3. 能力详解

### C01 本地工作区初始化

**目标**：让用户无需注册登录即可开始使用。

输入：

- workspace 名称；
- 本地路径；
- LLM Provider 配置；
- 隐私策略。

输出：

```text
.workspace/
├── jobpilot.db
├── files/
├── exports/
├── transcripts/
├── vector_index/
└── config.yaml
```

验收：

- 一条命令能启动；
- 用户能创建工作区；
- 数据默认本地保存；
- 重启后数据仍在。

---

### C02 Chatbox 对话入口

**目标**：让用户用自然语言完成求职任务。

核心交互：

```text
用户：这是我的简历，帮我看看怎么转前端。
Agent：我整理出了你的技能、项目和下一步建议。
```

要求：

- 支持上传文件；
- 支持粘贴 JD；
- 支持生成产物卡；
- 支持用户确认；
- 支持重新生成；
- 支持导出。

验收：

- 不需要进入复杂页面；
- 每轮任务结束有明确结果；
- 用户知道下一步做什么。

---

### C03 简历解析

输入：

- PDF；
- DOCX；
- Markdown；
- TXT；
- 粘贴文本。

输出：

- 教育经历；
- 工作经历；
- 项目经历；
- 技术技能；
- 非技术经历；
- 转行线索；
- 待确认事实。

验收：

- 能提取主要信息；
- 对无法判断的数字和经历标注待确认；
- 输出能被后续生成工具引用。

---

### C04 项目资料解析

输入：

- README；
- GitHub 项目描述；
- 用户粘贴的项目说明；
- Demo 说明；
- 技术栈。

输出：

- 项目卡；
- 简历 bullet 候选；
- 技术难点；
- 项目深挖问题；
- README 补强建议。

验收：

- 能把普通项目描述转成可面试表达；
- 能指出项目缺少部署、测试、架构、数据等证据；
- 不夸大项目规模。

---

### C05 职业事实库

事实类型：

- Experience；
- Project；
- Skill；
- Achievement；
- Education；
- Transition motivation；
- Soft skill；
- Metric。

字段：

- 内容；
- 来源；
- 置信度；
- 是否用户确认；
- 可见性；
- 适用场景。

验收：

- 生成简历时能引用事实；
- 不确定内容会被标记；
- 用户可以修改事实。

---

### C06 技能证据库

**目标**：避免“会 React”但没有项目证据。

每项技能关联：

- 项目；
- 代码/README/Demo；
- 具体实现；
- 面试可讲内容；
- 简历表达。

验收：

- 能指出“技能有证据/证据不足”；
- 能建议怎么补强；
- 能在 JD 匹配时使用。

---

### C07 JD 解析

输入：JD 文本。

输出：

- title；
- company；
- responsibilities；
- must-have；
- nice-to-have；
- tech stack；
- seniority guess；
- hidden expectations；
- interview focus。

验收：

- 能正确识别前端/后端/全栈等岗位重点；
- 能区分硬性要求和加分项；
- 能提取面试准备方向。

---

### C08 岗位适合度分析

输出不要只给分数，而要给：

```text
适合度：比较适合 / 可以尝试 / 先补强再投 / 暂不建议

为什么：
- 已匹配点
- 主要缺口
- 申请前建议
- 是否需要优化项目/简历
```

验收：

- 能帮用户决定是否申请；
- 能给投前 1-3 个行动；
- 语气低压力。

---

### C15-C17 实时面试提示

P0 可以分两层实现：

1. **文本实时助手**：用户粘贴/输入当前问题，系统立刻给提示。  
2. **语音实时助手**：麦克风/ASR adapter 提供 transcript，系统自动识别问题并提示。

提示内容：

```text
当前问题：讲一个你解决技术难题的经历。
建议用：TodoPlus 登录鉴权项目。
回答结构：
1. 先说问题
2. 讲你的实现
3. 讲一个取舍
4. 用结果收尾
提醒：不要说 OAuth，如果项目里没有。
```

验收：

- 能推荐项目；
- 能给结构；
- 能提醒时长；
- 默认不代答。

### C20 Agent Tool API

目标：所有核心能力都能独立调用。

工具输入输出必须：

- JSON Schema 化；
- 明确错误码；
- 产物可保存；
- 可被 Chatbox/CLI/MCP 调用。

验收：

- 至少 10 个核心工具可通过 API 调用；
- Chatbox 不直接写业务逻辑；
- 工具调用有日志。

## 6. MVP 最小验收闭环

```text
用户上传简历和项目
→ 系统生成职业事实库、技能证据库、项目卡
→ 用户粘贴 JD
→ 系统生成岗位适合度报告
→ 用户生成申请包
→ 系统导出 Markdown/PDF
→ 用户生成面试准备包
→ 用户模拟面试
→ 系统给实时提示或文本提示
→ 系统生成面试复盘和下一步计划
```

只要这个闭环跑通，即使前端极简，也算 MVP 成立。
